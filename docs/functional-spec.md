# 기능 명세서

## 1. 목적

사용자가 지역별 사건의 증거와 사투리 진술을 조사하고, AI 용의자에게 자유롭게 질문해 간접 단서를 얻은 뒤 범인을 추리하는 웹 게임을 제공한다.

주요 사용자는 회원가입한 플레이어다. 공개 사건 정보는 로그인 전에도 일부 조회할 수 있지만, 세션 생성 이후 기능은 인증이 필요하다.

## 2. 사용자 흐름

```text
회원가입·로그인 → 지역·사건·난이도 선택 → 세션 생성
→ 증거 열람·용의자 선택 → 자유 질문 심문 → 단서·기록 확인
→ 최종 추리 → 엔딩·점수 → 진행도·사투리 기록 확인
```

세션은 서버가 남은 질문, 용의자별 질문 횟수, 현재 감정, 만료와 상태 전이를 관리한다.

## 3. 핵심 기능

| 기능 | 설명 | 관련 화면/API |
|---|---|---|
| 인증·설정 | Supabase Auth 로그인과 프로필, BGM·효과음·텍스트 속도 저장 | 인증·설정, `/api/auth/*` |
| 사건 선택 | 지역별 공개 에피소드, 난이도, 용의자 공개 정보 조회 | 지역·에피소드, `/api/regions/*`, `/api/episodes/*` |
| 세션 | 난이도별 질문 수와 초기 증거·단서를 가진 서버 세션 생성 | 사건 수사, `/api/sessions/*` |
| 증거 열람 | 세션에서 사용 가능한 증거만 조회하고 열람 시각 기록 | 사건 기록, `/api/sessions/:id/evidence` |
| AI 심문 | 질문 유형을 분류하고 허용 fact만 사용한 사투리 구조화 답변 생성 | 심문, `/api/sessions/:id/interrogations` |
| 단서 해금 | 증거 열람·제시, 질문 유형, 용의자, fact 사용 등의 조건을 평가 | 심문·기록, `/api/sessions/:id/clues` |
| 사건 기록 | 증언, 타임라인, 관계, 메모와 획득 자료를 통합 조회 | 사건 기록, `/api/sessions/:id/records` |
| 최종 추리 | 선택한 용의자와 획득 단서를 서버에서 판정하고 점수 계산 | 추리, `/api/sessions/:id/deduction` |
| 엔딩·진행도 | 고정 엔딩과 선택적 보고서, 플레이 이력과 사투리 해금 제공 | 결과·프로필, `/api/progress/*` |

### 단서 평가 규칙

- 같은 `group_no`의 조건은 모두 충족해야 한다(AND).
- 서로 다른 그룹 중 하나만 완성돼도 해금된다(OR).
- `FACT_USED`는 최종 NPC 답변에 반영된 fact ID를 확인한다.
- `FACT_REVEALED`와 `CLAIM_RECORDED`는 각각 신규 공개와 NPC 주장을 별도로 기록한다.
- 세션별 단서 unique 제약으로 재평가해도 중복 획득하지 않는다.
- 단서 목록 조회 시 저장된 과거 심문도 재평가하되 질문 수, 메시지, 점수는 변경하지 않는다.

## 4. 예외 처리

| 상황 | 처리 |
|---|---|
| 인증 누락·다른 사용자 세션 | 401 또는 소유권을 노출하지 않는 404 오류 |
| 질문 수 소진·용의자별 한도 초과 | LLM 호출 전에 요청 차단 |
| 세션 만료·잘못된 상태 | 허용된 상태 전이 외 요청 차단 |
| 미획득 증거 제시 | 심문 요청 거부 |
| 중복 `requestId` | 기존 심문 결과 반환, 질문 재차감 방지 |
| LLM 실패·잘못된 fact | 제한된 재시도 후 안전한 오류 반환, 트랜잭션 미반영 |
| 프롬프트 탈취·범인 직접 공개 | 로컬 방어 응답으로 대체 |
| 단서 서버 오류 | 재시도 UI와 표준 API 오류 표시 |

## 5. 제한사항

- 실제 인증과 데이터 기능에는 유효한 Supabase 환경변수가 필요하다.
- 동적 심문·보고서에는 OpenAI API 키가 필요하다.
- Google OAuth는 구현되지 않았으며 UI에 준비 중으로 표시된다.
- 로컬 migration 전체 재현은 Docker Desktop과 Supabase CLI가 필요하다.
- 웹 앱만 제공하며 실시간 다중 사용자 기능이나 네이티브 앱은 포함하지 않는다.
